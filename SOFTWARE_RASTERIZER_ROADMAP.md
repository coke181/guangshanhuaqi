# Software Rasterizer Roadmap

这份 roadmap 按当前代码实际状态重写，目的不是继续堆零散功能点，而是把项目现状和下一阶段方向整理成一份更适合求职展示的说明。

---

## 1. 当前项目定位

当前项目已经不是“只会画三角形”的软光栅化练习，而是一个：

- 以 `C++` 为核心实现的 CPU 软件光栅化器
- 以 `Qt Widgets` 承担 UI、面板、视口交互和文件工作流
- 同时覆盖 `3D 渲染实验平台` 与 `2D 线稿图像处理` 两条能力线
- 已具备场景、对象、材质、灯光、调试视图、导出、测试和并行优化能力

如果写进简历，当前版本更适合描述为：

> 基于 C++ / Qt 实现的小型软件渲染实验平台，包含完整基础光栅化管线、对象级场景与材质编辑、资源资产化、OBJ 加载与场景序列化、多灯光与方向光阴影、真实 4x MSAA、tile 化多线程光栅化、调试视图体系，以及线稿图像处理工作流。

## 3. 当前确认还没做完、最值得继续补的内容

这一部分才是现在真正应该保留在 roadmap 里的内容。

### 3.1 灯光系统补完

当前还缺：

- ` 

为什么值得做：

- 这是当前图形算法深度上最明显的缺口
- 做完后灯光系统会从“够用”升级到“更完整的实时渲染实验平台”

### 3.2 场景层级与复用能力

当前还缺：

- 对象父子层级
- 本地 / 世界变换链
- 更正式的对象实例化 / 预制体复用

为什么值得做：

- 现在场景已经支持多对象与编辑，但还是“平铺对象列表”
- 补上层级后，编辑器结构会更像真正的 DCC / 引擎工具

### 3.3 更正式的材质系统上层

当前还缺：

- 材质类型与 shader 资产进一步解耦
- 更标准的材质参数块组织
- 共享材质浏览 / 资源面板 / 引用关系展示
- 贴图缩略图、缺失资源提示、重定向修复

为什么值得做：

- 底层材质与资源系统已经有了
- 现在更缺的是“编辑器层的资源工作流完整度”

### 3.4 自动化回归与结果对比工具

当前还缺：

- 基准图管理
- 图像 diff 输出
- CLI 场景回归入口
- benchmark 结果持久化与报表

为什么值得做：

- 现在已经有测试、截图、调试视图、benchmark 基础
- 再往前一步就是正式工程化能力，而不是只在 UI 里手测

### 3.5 更完整的画质链

当前还缺：

- 环境光照 / IBL
- 更完整的透明排序与半透明质量策略
- 屏幕后处理增强
  - `FXAA / TAA` 方向
  - bloom
  - color grading LUT

为什么值得做：

- 当前项目在“基础渲染器”层已经比较完整
- 后续更适合往“画质”和“效果链”方向拉开差异化

### 3.6 动画与可演示性

当前还缺：

- 更正式的对象动画系统
- 相机轨迹 / 关键帧
- 场景演示 preset 的镜头编排

为什么值得做：

- 这会明显提升项目展示效果
- 也方便做 benchmark、导出序列和求职作品录屏

## 7. 一句话结论

当前项目已经不缺“基础软光栅化功能”，真正还值得继续补的是：

- 更完整的灯光与阴影
- 更正式的场景层级与复用
- 更成熟的资源 / 材质工作流
- 自动化回归与性能报告
- 更强的画质链与演示能力

后续 roadmap 应该围绕这些方向推进，而不是再把已经做完的基础功能继续挂在待办里。
